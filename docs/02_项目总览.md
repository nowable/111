## 一、这台车要做什么

一台基于地平线 **RDK X5** 的 ROS2 自主小车，在**医院主题**场地里跑完整一圈任务：

```
   ┌─────────────────────────────────────────────────────────┐
   │  A 区（浅蓝大厅）          B 区          C 区（环道）     │
   │                                                          │
   │  P点发车 ──→ 避深蓝三角 ──→ 扫二维码 ─→ 黄色通道 ─→ 绕诊室│
   │   ↑                         （定左右）   (1m直道)  一圈   │
   │   │                                                 │    │
   │   └──────────── 返回 P点 ←──── 拍橙色图文牌送AI ←────┘    │
   └─────────────────────────────────────────────────────────┘
```

逐段说明：

1. **发车**：从 P 点（A 区浅蓝大厅）出发。
2. **避障**：避开地上的**深蓝色三角路障**。
3. **扫码定向**：在右侧扫一个**二维码**，码里写了走顺时针还是逆时针（= 右环 / 左环）。
4. **过通道**：沿 B 区 **1 米宽的黄色通道**直行。
5. **绕环**：进 C 区，沿 **0.5 米宽的黄色环道**绕中央的绿色诊疗室（禁止进入）走一圈。
6. **识别送 AI**：环道拐角有**橙色图文牌**，拍下来发给多模态大模型（阿里百炼 qwen-vl）识别内容。
7. **返航**：原路返回 P 点。

导航主要靠**看地面的黄色线循迹**，辅以里程计 + IMU 航向兜底（计圈、返航）。

## 二、整体怎么运转

车上跑一个主程序 `auto_mission`，它是一个**状态机**——按固定顺序一个状态接一个状态地走：

```
IDLE（待命）
  → PREFLIGHT（起飞前自检：相机/电压/底盘都正常吗）
  → DRIVE_TO_QR（低速往前开，边躲深蓝障碍，直到看见二维码板）
  → SCAN_QR（停下，解码二维码）
  → DECIDE_DIRECTION（根据码内容决定走左环还是右环）
  → CORRIDOR_FOLLOW（沿黄色通道直行过 B 区）
  → LOOP_LEFT / LOOP_RIGHT（沿黄色环道绕一圈，沿途拍橙色牌）
  → CAPTURE_TARGET_IMAGE（挑最好的一张目标图）
  → CALL_LLM_API（发给 AI 识别，结果发布到话题）
  → RETURN_HOME（按来时记录的轨迹原路返回）
  → STOP → DONE（任务完成）
```

每一步都会在终端打印 `STATE: xxx` 和结构化的 `EVENT {...}` 日志，方便事后复盘。

**三个关键能力**支撑这套流程：
- **看颜色**（`vision_detector`）：用 OpenCV 按 HSV 区分深蓝障碍、黄色线、橙色牌、绿色禁区；可选叠加板载 YOLO。
- **知道自己走了多远、转了多少度**（`motion_state`）：融合里程计 + IMU，用于判断"环绕够一圈没"和"返航该怎么倒回去"。
- **循迹**（`lane_follow`）：盯住画面下方的黄线，算出该往左还是往右打方向（PD 控制）。

## 三、怎么上手跑

> 真车是 SSH 里的 `rdkx5`。所有命令先进环境：
> `source /opt/ros/humble/setup.bash && source /root/dev_ws/install/setup.bash`
> 改完代码先编译：`cd /root/dev_ws && colcon build --symlink-install --packages-select origin_competition_auto`

由安全到危险，分四级测试：

### 1) 纸上推演（不碰任何硬件）
```bash
ros2 run origin_competition_auto auto_mission --config <配置> --dry-run \
  --mock-qr-content '{"direction":"left"}'
```
打印完整状态流和每步动作，确认逻辑没问题。最安全，随便跑。

### 2) 静态自检（车不动）
```bash
ros2 run origin_competition_auto auto_mission --config <配置> --no-motion \
  --start-state CAPTURE_TARGET_IMAGE
```
`--no-motion` 屏蔽所有轮子指令，可以验证相机、二维码解码、AI 调用这些"不动腿"的部分。`--start-state` 可以从任意状态开始测。

### 3) 架空低速（车架起来，轮子空转）
用 `bench_motion` 运行档，验证电机方向、循迹打方向对不对。

### 4) 地面慢速（真跑）
用 `field_low_speed` 运行档，保守速度在场地里实跑。

### 一键编排
`competition_run` 把"自检 → 启 YOLO → 跑任务 → 回放复盘"串成一条命令，按运行档（profile）执行，所有产物落到 `runs/<时间戳>/` 目录。

## 四、AI 识别（大模型）怎么配

用阿里百炼 DashScope 的视觉模型 `qwen-vl-plus`（OpenAI 兼容接口）。

- **默认是关的**（`llm_mode=placeholder`），不会真发网络请求，避免误触发。要真调用时加 `--llm-mode openai-compatible`。
- **密钥**存在板子 `/root/dev_ws/.dashscope_key`（权限 600，不进代码库）。跑之前先注入环境变量：
  ```bash
  export DASHSCOPE_API_KEY=$(cat /root/dev_ws/.dashscope_key)
  ```
- 识别结果会发布到 ROS 话题 `/competition/llm_result`，也打印在日志里。
- 图片在发送前会自动缩小（最长边 1024，JPEG 质量 85），省流量也防超时。

> 注意：`qwen-vl-max-latest` 这个模型当前密钥没权限（会报 403），用 `qwen-vl-plus`。

## 五、所有参数都在一个文件里

这是本项目最重要的设计：**几乎所有能调的东西都集中在 `config/mission_defaults.json`**——速度、各种颜色的 HSV 范围、每个阶段的超时、循迹的灵敏度、返航策略……场地日只改这个文件的数字，不用碰代码。

几个最常调的：

| 想调什么 | 改哪个键 |
|----------|---------|
| 车开多快 | `cruise_linear` / `lane_base_linear`（默认很慢 0.03，安全优先） |
| 黄线认不准 | `color_profiles.yellow` 的 HSV 范围（家里光线和场地不同，常需收紧） |
| 循迹蛇行或反应慢 | `lane_kp`（比例）/ `lane_kd`（阻尼） |
| 二维码停太远/太近 | `qr_min_area_ratio`（越大停越近） |
| 绕环不到一圈就停 | `loop_complete_fraction`（占 360° 的比例） |
| 避障太敏感/迟钝 | `obstacle_danger_area_ratio` |

> **重要**：`auto_mission` 必须带 `--config` 参数指到这个文件才会加载，否则用代码里的内置默认值（那套默认 AI 密钥变量名是给 OpenAI 的，不是 DashScope）。

## 六、配套工具链

除了主程序，项目还有一整套围绕"采数据→调参数→验证→交付"的工具（都是 `ros2 run origin_competition_auto <名>`）：

- **采数据**：`field_session`（建采集会话、生成清单和进度跟踪）+ `dataset_capture`（实际抓帧存图，可自动粗标注）。
- **调视觉**：`lane_debug`（单张图量化任意 `color_profiles` 颜色的 `mask_area_ratio`）+ `vision_debug`（单张图看深蓝障碍/橙色牌检测效果）+ `vision_tune`（旧版黑障碍/白牌参数网格搜索，不会自动产出四颜色 `color_profiles`）。
- **离线验证**：`mission_replay`（不用车、不用 ROS，对存下来的图重跑整条感知决策链）。
- **复盘固化**：`field_data_review`（把审计+回放+调优合成一次复盘，给出"就绪度"评估和推荐的参数补丁）+ `apply_review_recommendations`（把补丁安全地写回配置，带白名单校验和备份）。
- **赛前检查**：`system_check`（只读检查服务/话题/相机/电压）。
- **运动标定**：`motion_calibration`（限幅发轮子指令，校准电机）。
- **YOLO**：`yolo_pipeline`（准备/导出/固化训练产物）。
- **验收交付**：`solution_audit`（生成验收矩阵）+ `handoff_bundle`（打包交付）。

详细中文操作说明在 `origin_competition_project/docs/` 目录（赛前运行手册、数据采集说明、视觉调优说明等十多份）。

## 七、出问题怎么排查

| 现象 | 可能原因 / 怎么办 |
|------|------------------|
| 车不动、`/cmd_vel` 没反应 | 底盘串口节点可能因 USB 抖动崩了 → `restart origincar-base.service` |
| 从 P 出发只走几厘米就停 | 如果用 `competition_run --profile field_low_speed`，这是 2s 接近测试档；完整跑用 `auto_mission --config <cfg>`。如果直接跑 `auto_mission` 仍这样，看终端是否停在 `SCAN_QR` / `QR scan timeout` |
| 循迹乱打方向 / 把别的东西当黄线 | 黄色 HSV 范围太松 → 用 `dataset_capture` 抓黄线正/负样本，再用 `lane_debug --color yellow --config <cfg>` 看 `mask_area_ratio`，手动收紧 `color_profiles.yellow` |
| 绕环不停 / 提前停 | `loop_complete_fraction` 或 IMU 漂移 → 检查 `motion_state_debug` 输出的 integrated_yaw |
| AI 报 403 access_denied | 模型没权限 → 确认用 `qwen-vl-plus`；确认 `DASHSCOPE_API_KEY` 已 export |
| AI 报 missing_env | 没带 `--config` 或没 export 密钥 → 两者都要 |
| 二维码扫不出 | 光照/距离 → 用 `qr_capture` 单独调；调 `qr_min_area_ratio` 让车靠更近 |
| 想看任务到底走到哪步 | 看终端 `STATE:` 日志，或 `runs/<时间戳>/mission_output.log` |

排查时善用各个 `_debug` 工具单独验证某一环，比直接整车跑快得多。
