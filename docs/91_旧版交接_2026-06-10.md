## 比赛是什么

这是一个基于地平线 RDK X5 / OriginCar 的医院主题自主车场地任务。车需要在固定尺寸的医院场景图中，从 P 点自主发车，识别障碍、二维码、黄色通道、绿色禁区和图文标记牌，最后返回 P 点。项目代码包名是 `origin_competition_auto`，当前实现重点是用 ROS2 节点控制小车完成完整比赛状态机。

比赛不是普通巡线单项，也不是只跑固定路线。它同时包含：

1. 固定场地图路线理解。
2. A 区大厅避障和到达二维码。
3. 二维码内容解析，决定后续 C 区绕圈方向。
4. B 区黄色通道通行。
5. C 区诊疗室外围黄色环道绕行，禁止进入绿色诊疗室区域。
6. 经过指定图文标记牌时拍照，调用多模态 LLM 生成文本描述。
7. 从 C/B 区返回 P 点。
8. 全程限时内完成，并避免碰墙、碰障碍、压绿色禁区。

比赛每轮时间上限是 **180 秒**。当前程序配置里的 `max_runtime` 设计为 **175 秒**，目的是比规则上限提前 5 秒强制停车，避免超时后车继续动。看到日志里：

```text
failure_reason: max runtime exceeded
```

不一定代表程序崩溃，通常只是运行时间碰到 `max_runtime`。如果是正式比赛，全程必须在 180 秒以内完成；如果是调试命令里手动传了 `--max-runtime 12` 或 `--max-runtime 80`，则会按命令里的短时间提前停。

## 场地分区

整张场地可以理解为三个主区域：A 区大厅、B 区通道、C 区诊疗室。代码中的 `STATE_ZONES`、地图航点和运行日志也基本按这三个区域组织。

### A 区：浅蓝色大厅

图中文字：

```text
区域A：浅蓝色区域（大厅）
面积：长*宽（5米*2米）
区域内有：发车点/停车点、任务发布点以及若干个非固定位置的障碍物（黑色三角）
```

实际任务含义：

- A 区是发车区和返航终点区。
- `P` 是发车点/停车点，位于图左下角。
- 车从 `P` 点出发，初始朝向允许自定义，但必须和代码 `map_start_pose.heading_deg` 一致。
- A 区内有若干障碍物，图上画的是黑色三角。实际现场用户提到的是深蓝色障碍桶/障碍物，因此当前视觉主兜底是 `dark_blue` HSV，YOLO 只是辅助。
- A 区地面有黑色虚线。用户的策略已经从纯视觉触发改成“内置地图路线 + 视觉/QR/障碍辅助”，但今天外场暴露固定路线还需要继续调。
- A 区右侧靠近二维码放置区，有二维码和图文标记牌示意。

当前代码相关状态：

```text
DRIVE_TO_QR
```

当前路线名：

```text
A_TO_QR
```

当前日志重点：

```text
MAP_ROUTE_START name=A_TO_QR
MAP_TARGET waypoint=A_DASH_01 dist=... heading_err=...
QR_CANDIDATE_DURING_DRIVE ...
QR_BOARD_REACHED ...
QR_CONTENT ...
```

必须注意：

- `QR_BOARD_REACHED` 不等于真正扫码成功。
- 真正扫码要看 `QR_CONTENT` 非空，并且 `DIRECTION_SOURCE` 是 QR 解析来源，而不是 `force_direction`。
- 今天外场发现 OpenCV QR candidate points 会远距离误触发，所以 QR 判定必须整改。

## 场地图符号说明

图右下角给出了符号含义：

```text
黄色菱形：二维码放置区
橙色小牌：图文标记牌
A/B/C 方框：物品放置区
黑色三角：障碍物
```

结合当前代码和用户现场反馈，需要这样理解：

- 二维码放置区：用于读取任务方向，内容应能解析出 `left/right` 或中文左/右、顺/逆时针。
- 图文标记牌：车经过对应位置时拍照，送多模态 LLM，生成中文描述。
- 物品放置区 A/B/C：图中在上方标注，可能是物品或任务牌位置；当前代码主要没有做精细物品搬运，只做图文牌识别。
- 障碍物：图示为黑色三角，但实际外场用户采集的是障碍桶/深蓝障碍。下一个 AI 不能只按黑色三角写死视觉规则。

## 完整任务流程

当前期望流程：

```text
P 点发车
  -> A 区沿黑色虚线/内置地图路线接近二维码
  -> 途中识别并避开深蓝障碍物
  -> 到二维码处停车扫码
  -> 解析二维码方向 left/right
  -> 根据方向决定 C 区绕圈方向
  -> 从二维码位置进入 B 区黄色通道
  -> B 区沿黄色通道进入 C 区
  -> C 区按二维码方向绕黄色环道一圈
  -> 在当前方向对应的蓝圈/图文牌窗口抓拍
  -> 调用 LLM 生成图文牌中文描述
  -> 沿第二条黑色虚线或地图返航路线回 P 点
  -> STOP / DONE
```

当前代码状态机：

```text
IDLE
  -> PREFLIGHT
  -> DRIVE_TO_QR
  -> SCAN_QR
  -> DECIDE_DIRECTION
  -> CORRIDOR_FOLLOW
  -> LOOP_LEFT / LOOP_RIGHT
  -> CAPTURE_TARGET_IMAGE
  -> CALL_LLM_API
  -> RETURN_HOME
  -> STOP
  -> DONE
```

调试时常用从中间状态启动，例如：

```bash
ros2 run origin_competition_auto auto_mission --config $CFG --start-state DRIVE_TO_QR
ros2 run origin_competition_auto auto_mission --config $CFG --start-state SCAN_QR --no-motion
ros2 run origin_competition_auto auto_mission --config $CFG --start-state CORRIDOR_FOLLOW
```

但正式比赛应该从 `IDLE` 或 `DRIVE_TO_QR` 的完整流程开始，且必须带 `--config`。

## 比赛限时与程序超时

规则上限：

```text
每轮比赛 180 秒
```

程序默认：

```json
"max_runtime": 175.0
```

含义：

- `max_runtime=175` 是安全强停，不是规则时间。
- 正式比赛剩余 5 秒给停车、裁判判定和异常缓冲。
- 调试时用户经常传 `--max-runtime 12`、`--max-runtime 80`，这些会覆盖配置。
- 看到 `failure_reason: max runtime exceeded` 时，先看命令里是否人为设了短时间。

其他关键超时：

- `drive_to_qr_timeout`：A 区接近二维码阶段超时。
- `scan_qr_timeout`：静态扫码阶段超时。
- `map_route_timeout`：地图路线超时。
- `shunt_turn_timeout`：阿克曼底盘 shunt 近似原地转向超时。
- `map_return_timeout`：地图返航超时。

今天外场曾经出现：

```text

```

这不是任务总超时，而是 `shunt_turn_timeout=45` 不够完成约 147 度对齐。是否放宽要等 QR 到达判定修好后再测，不能在 QR 误触发情况下盲目调。

## 得分风险与失败条件

下一个 AI 调整代码时，要把比赛风险按优先级考虑：

1. 安全停车优先于继续任务。
2. 撞墙、碰障碍、驶入绿色诊疗室都会严重扣分或导致失败。
3. 误判二维码到达会导致后续全流程错位。
4. C 区黄色环道只有 0.5 米，不能靠高速硬冲。
5. `--force-direction right/left` 只适合调试，不能证明真实扫码成功。
6. 不要用远距离 QR candidate points 作为到达依据。
7. 不要直接覆盖板子现场 HSV。
8. 每次代码同步前备份板子源码和配置。

### Step 2: 正式整理 QR 判定

必须改成：

- 行进阶段只有 QR 角点但没有 `qr.text` 时，不能 `QR_BOARD_REACHED`。
- 可打印：

```text
QR_BOARD_CANDIDATE_IGNORED_NO_DECODE area=... offset=...
```

- 只有真的解码，且方向被确认，才允许 `DECIDE_DIRECTION`。
- 如果希望靠近后停下来扫码，需要新增更严格条件，例如：
  - 面积阈值更高；
  - 连续 N 帧候选；
  - 候选框长宽比合理；
  - 候选框 y 位置合理；
  - 或到达地图 QR 航点附近后才允许 board reached。

当前外场事实证明：单靠 OpenCV `points + area + center` 会远距离误触发。

### Step 3: 正式整理起步保护

当前本地版本有字段，板子热补版本不完整。建议正式逻辑为：

- guard 只在路线开始前 1 到 2 秒内生效；
- 或只在 `pose` 接近 `P_START` 时生效；
- 不要在运行几十秒后仍对 `A_DASH_01` 做 start guard；
- 配置默认可关闭：

```json
"map_start_heading_guard_enabled": false,
"map_start_heading_guard_rad": 0.6
```

如果保留，日志必须打印当前配置值，便于确认配置生效。

## 下一个 AI 不要做的事

- 不要直接覆盖板子 `mission_defaults.json`，会丢现场 HSV。
- 不要把 `map_max_linear` 改到很大；`10` 是危险值，单位是 m/s。
- 不要把 `--force-direction right` 当成扫码成功；它只是在跳过二维码方向解析。
- 不要看到 `QR_BOARD_REACHED` 就认为扫到二维码；必须看 `QR_CONTENT` 和 `DIRECTION_SOURCE`。
- 不要继续现场完整跑任务，先离线整理热补和照片。
- 不要把 `MAP_START_HEADING_MISMATCH` 当路线根因；它现在是热补不完整导致的副作用。
